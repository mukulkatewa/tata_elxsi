# Requirements Document

## Introduction

AutoForge Phase 2 extends the existing RAG knowledge base system with an LLM-powered multi-agent code generation pipeline for automotive embedded software development. The system orchestrates specialized AI agents (Dev Agent and QA Agent) to automatically generate MISRA-compliant C++ services and corresponding test code based on natural language requirements. By combining the automotive domain knowledge from Phase 1's RAG system with large language model capabilities, Phase 2 enables rapid prototyping of Software Defined Vehicle (SDV) components with built-in safety standards compliance and automated testing.

## Glossary

- **Dev_Agent**: The AI agent responsible for generating production C++ service code (.cpp and .h files) based on requirements and RAG context
- **QA_Agent**: The AI agent responsible for generating C++ test code with mock data providers and assertion-based validation
- **Orchestrator**: The component that coordinates the Dev_Agent and QA_Agent, manages file I/O, and handles the complete code generation workflow
- **LLM_Client**: The OpenAI API client instance used to communicate with language models
- **Service**: A self-contained C++ class that processes vehicle signals and generates alerts or outputs
- **Prompt_Template**: A text file containing system instructions and placeholder variables for LLM prompts
- **RAG_Context**: The automotive domain knowledge (VSS signals, MISRA rules, ASPICE items) retrieved from the Phase 1 knowledge base
- **Service_Name**: A snake_case identifier derived from the first 3-4 meaningful words of a requirement
- **Output_Directory**: The file system location where generated code and metadata are saved (autoforge/outputs/[service_name]/)
- **Metadata_File**: A JSON file containing generation details including requirement, service name, VSS signals used, timestamp, and model name

## Requirements

### Requirement 1: Environment Configuration Management

**User Story:** As a developer, I want centralized configuration management with environment variables, so that I can securely configure API keys and system settings without hardcoding values.

#### Acceptance Criteria

1. THE Configuration_Module SHALL load environment variables from a .env file using python-dotenv
2. THE Configuration_Module SHALL provide settings for LLM_PROVIDER with default value "openai"
3. THE Configuration_Module SHALL provide settings for OPENAI_API_KEY loaded from environment
4. THE Configuration_Module SHALL provide settings for MODEL_NAME with default value "gpt-4o-mini"
5. THE Configuration_Module SHALL provide settings for MAX_TOKENS with default value 2048
6. THE Configuration_Module SHALL provide settings for CHROMA_DB_PATH with default value "autoforge/data/chroma_db"
7. THE Configuration_Module SHALL provide settings for OUTPUTS_DIR with default value "autoforge/outputs"
8. THE Configuration_Module SHALL provide a get_llm_client function that returns an initialized OpenAI client instance
9. WHEN OPENAI_API_KEY is not set in environment, THE Configuration_Module SHALL raise a clear error message indicating the missing API key

### Requirement 2: Dev Agent Prompt Template

**User Story:** As a code generation system, I want a structured prompt template for the Dev Agent, so that generated C++ services follow consistent automotive embedded software patterns.

#### Acceptance Criteria

1. THE Dev_Agent_Prompt_Template SHALL be stored at prompts/dev_agent_prompt.txt
2. THE Dev_Agent_Prompt_Template SHALL include placeholders for {requirement}, {vss_context}, {misra_context}, and {aspice_context}
3. THE Dev_Agent_Prompt_Template SHALL instruct the LLM to act as a senior automotive embedded software engineer
4. THE Dev_Agent_Prompt_Template SHALL specify generation of self-contained C++ services with both .cpp and .h file content
5. THE Dev_Agent_Prompt_Template SHALL require use of only VSS signals provided in the context
6. THE Dev_Agent_Prompt_Template SHALL require MISRA rule compliance with inline comments in format "// MISRA Rule X.X: [reason]"
7. THE Dev_Agent_Prompt_Template SHALL specify class structure with constructor, init method, process method accepting map<string,float>, and getAlerts method
8. THE Dev_Agent_Prompt_Template SHALL require signal validation against VSS min and max ranges
9. THE Dev_Agent_Prompt_Template SHALL require file headers containing service name, requirement, MISRA compliance statement, and "generated by AutoForge" attribution
10. THE Dev_Agent_Prompt_Template SHALL instruct the LLM to return ONLY raw C++ code without markdown formatting or explanatory text

### Requirement 3: QA Agent Prompt Template

**User Story:** As a code generation system, I want a structured prompt template for the QA Agent, so that generated test code validates service behavior with automotive-specific test scenarios.

#### Acceptance Criteria

1. THE QA_Agent_Prompt_Template SHALL be stored at prompts/qa_agent_prompt.txt
2. THE QA_Agent_Prompt_Template SHALL include placeholders for {requirement}, {generated_code}, and {service_name}
3. THE QA_Agent_Prompt_Template SHALL instruct the LLM to act as an automotive QA engineer
4. THE QA_Agent_Prompt_Template SHALL specify generation of C++ test files using a simple mock approach without external testing frameworks
5. THE QA_Agent_Prompt_Template SHALL require a main function with assert statements for validation
6. THE QA_Agent_Prompt_Template SHALL require test cases for normal values, boundary values, out-of-range values, and missing signals
7. THE QA_Agent_Prompt_Template SHALL require print statements in format "PASS: [test name]" or "FAIL: [test name]"
8. THE QA_Agent_Prompt_Template SHALL require inclusion of a MockDataProvider class
9. THE QA_Agent_Prompt_Template SHALL instruct the LLM to return ONLY raw C++ code without markdown formatting

### Requirement 4: Dev Agent Code Generation

**User Story:** As an automotive software developer, I want an AI agent that generates MISRA-compliant C++ services from natural language requirements, so that I can rapidly prototype vehicle software components.

#### Acceptance Criteria

1. THE Dev_Agent SHALL accept a requirement string as input to its generate method
2. WHEN generate is called, THE Dev_Agent SHALL retrieve automotive context using RAGRetriever.retrieve_context
3. WHEN generate is called, THE Dev_Agent SHALL load the dev_agent_prompt.txt template and fill all placeholders with requirement and RAG context
4. WHEN generate is called, THE Dev_Agent SHALL make an OpenAI API call with the filled prompt
5. WHEN generate is called, THE Dev_Agent SHALL parse the LLM response to extract C++ code
6. THE Dev_Agent SHALL derive a service_name in snake_case format from the first 3 to 4 meaningful words of the requirement
7. THE Dev_Agent SHALL return a dictionary containing service_name, header_code, source_code, full_code, requirement, and vss_signals_used keys
8. THE Dev_Agent SHALL extract vss_signals_used by scanning generated code for "Vehicle." pattern occurrences
9. WHEN generating code, THE Dev_Agent SHALL print progress message "🤖 Dev Agent generating [service_name]..."
10. IF the OpenAI API call fails, THE Dev_Agent SHALL raise an exception with the API error details

### Requirement 5: QA Agent Test Generation

**User Story:** As a quality assurance engineer, I want an AI agent that generates comprehensive test code for generated services, so that I can validate service behavior across normal and edge case scenarios.

#### Acceptance Criteria

1. THE QA_Agent SHALL accept requirement, generated_code, and service_name as inputs to its generate_tests method
2. WHEN generate_tests is called, THE QA_Agent SHALL load the qa_agent_prompt.txt template and fill all placeholders
3. WHEN generate_tests is called, THE QA_Agent SHALL make an OpenAI API call with the filled prompt
4. THE QA_Agent SHALL return a dictionary containing test_code, service_name, and test_file_name keys
5. THE QA_Agent SHALL format test_file_name as "test_[service_name].cpp"
6. WHEN generating tests, THE QA_Agent SHALL print progress message "🧪 QA Agent generating tests for [service_name]..."
7. IF the OpenAI API call fails, THE QA_Agent SHALL raise an exception with the API error details

### Requirement 6: Multi-Agent Orchestration

**User Story:** As a system user, I want an orchestrator that coordinates code generation agents and manages file outputs, so that I can generate complete service implementations with a single function call.

#### Acceptance Criteria

1. THE Orchestrator SHALL initialize RAGRetriever, Dev_Agent, and QA_Agent instances during construction
2. THE Orchestrator SHALL provide a run method that accepts a requirement string
3. WHEN run is called, THE Orchestrator SHALL derive service_name from the requirement
4. WHEN run is called, THE Orchestrator SHALL create output directory at outputs/[service_name]/ with parents and exist_ok flags
5. WHEN run is called, THE Orchestrator SHALL call Dev_Agent.generate with the requirement
6. WHEN run is called, THE Orchestrator SHALL save generated C++ code to outputs/[service_name]/[service_name].cpp
7. WHEN run is called, THE Orchestrator SHALL call QA_Agent.generate_tests with requirement, generated code, and service name
8. WHEN run is called, THE Orchestrator SHALL save test code to outputs/[service_name]/test_[service_name].cpp
9. WHEN run is called, THE Orchestrator SHALL save metadata.json containing requirement, service_name, vss_signals_used, timestamp, and model_used fields
10. WHEN run is called, THE Orchestrator SHALL print a final summary with file paths for generated code, test code, and metadata
11. THE Orchestrator SHALL return a dictionary containing all outputs and file paths
12. IF QA_Agent.generate_tests fails, THE Orchestrator SHALL save the generated service code and metadata, log the QA error, and continue execution

### Requirement 7: Interactive Mode

**User Story:** As a developer, I want an interactive mode that continuously accepts requirements, so that I can generate multiple services in a single session.

#### Acceptance Criteria

1. THE Orchestrator SHALL provide a run_interactive method
2. WHEN run_interactive is called, THE Orchestrator SHALL display a welcome message
3. WHILE run_interactive is active, THE Orchestrator SHALL prompt the user for a requirement input
4. WHEN the user enters a requirement, THE Orchestrator SHALL call the run method with that requirement
5. WHEN the user enters "quit" or "exit", THE Orchestrator SHALL terminate the interactive loop
6. WHILE run_interactive is active, THE Orchestrator SHALL handle errors gracefully and continue prompting for new requirements

### Requirement 8: Demo Mode Execution

**User Story:** As a hackathon participant, I want a demo mode with preset requirements, so that I can quickly showcase the system capabilities without manual input.

#### Acceptance Criteria

1. THE Entry_Point_Script SHALL be named run_phase2.py
2. THE Entry_Point_Script SHALL support a --demo command line flag
3. WHEN run with --demo flag, THE Entry_Point_Script SHALL execute 2 preset requirements without user interaction
4. THE first preset requirement SHALL be "Create a tyre pressure monitoring service that alerts when pressure drops below 28 PSI on any wheel"
5. THE second preset requirement SHALL be "Create a battery state of charge service that monitors EV range and triggers low battery warnings below 20%"
6. WHEN run without --demo flag, THE Entry_Point_Script SHALL call orchestrator.run_interactive
7. THE Entry_Point_Script SHALL import AutoForgeOrchestrator and handle initialization

### Requirement 9: Dependency Management

**User Story:** As a developer setting up the project, I want clear dependency specifications, so that I can install all required packages for Phase 2 functionality.

#### Acceptance Criteria

1. THE requirements.txt file SHALL include the openai package
2. THE requirements.txt file SHALL include the python-dotenv package
3. THE requirements.txt file SHALL preserve all Phase 1 dependencies (chromadb, sentence-transformers, langchain, langchain-community)

### Requirement 10: API Error Handling

**User Story:** As a system operator, I want robust error handling for LLM API calls, so that I receive clear feedback when API requests fail.

#### Acceptance Criteria

1. WHEN Dev_Agent makes an OpenAI API call, THE Dev_Agent SHALL wrap the call in a try-except block
2. WHEN QA_Agent makes an OpenAI API call, THE QA_Agent SHALL wrap the call in a try-except block
3. IF an OpenAI API call fails with authentication error, THE system SHALL raise an exception indicating invalid or missing API key
4. IF an OpenAI API call fails with rate limit error, THE system SHALL raise an exception indicating rate limit exceeded
5. IF an OpenAI API call fails with network error, THE system SHALL raise an exception indicating connection failure
6. IF an OpenAI API call fails with any error, THE system SHALL include the original error message in the raised exception

### Requirement 11: File System Safety

**User Story:** As a system administrator, I want safe file system operations, so that the system handles missing directories and prevents data loss.

#### Acceptance Criteria

1. WHEN creating output directories, THE Orchestrator SHALL use Path.mkdir with parents=True and exist_ok=True flags
2. WHEN saving generated files, THE Orchestrator SHALL ensure the parent directory exists before writing
3. IF a file write operation fails, THE Orchestrator SHALL raise an exception with the file path and error details
4. THE system SHALL use pathlib.Path for all file system operations to ensure cross-platform compatibility

### Requirement 12: Service Name Derivation

**User Story:** As a code organization system, I want consistent service naming conventions, so that generated files follow predictable patterns.

#### Acceptance Criteria

1. WHEN deriving service_name from a requirement, THE system SHALL extract the first 3 to 4 meaningful words
2. WHEN deriving service_name, THE system SHALL convert the extracted words to snake_case format
3. WHEN deriving service_name, THE system SHALL exclude common stop words like "create", "a", "the", "that", "when"
4. THE derived service_name SHALL contain only lowercase letters, numbers, and underscores
5. THE derived service_name SHALL be used consistently for directory names, file names, and class names

### Requirement 13: VSS Signal Extraction

**User Story:** As a traceability system, I want to track which VSS signals are used in generated code, so that I can maintain signal usage documentation.

#### Acceptance Criteria

1. WHEN Dev_Agent generates code, THE Dev_Agent SHALL scan the generated code for VSS signal references
2. THE Dev_Agent SHALL identify VSS signals by searching for the pattern "Vehicle." followed by signal path components
3. THE Dev_Agent SHALL extract all unique VSS signal names found in the generated code
4. THE Dev_Agent SHALL include the list of extracted signals in the vss_signals_used field of the return dictionary
5. THE vss_signals_used list SHALL be saved in the metadata.json file

### Requirement 14: Metadata Persistence

**User Story:** As a project manager, I want generation metadata saved with each service, so that I can track requirements, timestamps, and model versions.

#### Acceptance Criteria

1. FOR EACH generated service, THE Orchestrator SHALL create a metadata.json file in the service output directory
2. THE metadata.json file SHALL include a requirement field containing the original requirement string
3. THE metadata.json file SHALL include a service_name field
4. THE metadata.json file SHALL include a vss_signals_used field containing the list of VSS signals referenced in the code
5. THE metadata.json file SHALL include a timestamp field in ISO 8601 format
6. THE metadata.json file SHALL include a model_used field containing the LLM model name
7. THE metadata.json file SHALL be formatted with indentation for human readability

### Requirement 15: Code Parsing and Extraction

**User Story:** As a code generation system, I want to reliably extract C++ code from LLM responses, so that I can handle various response formats including markdown code blocks.

#### Acceptance Criteria

1. WHEN parsing LLM responses, THE Dev_Agent SHALL handle responses containing markdown code blocks with ```cpp delimiters
2. WHEN parsing LLM responses, THE Dev_Agent SHALL handle responses containing raw C++ code without markdown formatting
3. WHEN parsing LLM responses, THE Dev_Agent SHALL strip leading and trailing whitespace
4. WHEN parsing LLM responses, THE QA_Agent SHALL apply the same parsing logic as Dev_Agent
5. IF an LLM response contains multiple code blocks, THE system SHALL extract and concatenate all blocks

### Requirement 16: Progress Feedback

**User Story:** As a system user, I want real-time progress feedback during code generation, so that I understand what the system is doing at each step.

#### Acceptance Criteria

1. WHEN Dev_Agent starts generating code, THE Dev_Agent SHALL print a progress message including the service name
2. WHEN QA_Agent starts generating tests, THE QA_Agent SHALL print a progress message including the service name
3. WHEN Orchestrator completes generation, THE Orchestrator SHALL print a summary including paths to all generated files
4. THE progress messages SHALL use emoji indicators (🤖 for Dev Agent, 🧪 for QA Agent) for visual clarity
5. THE summary message SHALL include the service name, code file path, test file path, and metadata file path

### Requirement 17: Graceful Degradation

**User Story:** As a system operator, I want the system to save partial results when components fail, so that I don't lose generated code if test generation fails.

#### Acceptance Criteria

1. IF QA_Agent.generate_tests raises an exception, THE Orchestrator SHALL catch the exception
2. WHEN QA_Agent fails, THE Orchestrator SHALL save the generated service code from Dev_Agent
3. WHEN QA_Agent fails, THE Orchestrator SHALL save the metadata.json file with available information
4. WHEN QA_Agent fails, THE Orchestrator SHALL log the error message to console
5. WHEN QA_Agent fails, THE Orchestrator SHALL include an error field in the returned dictionary indicating test generation failure
6. WHEN QA_Agent fails, THE Orchestrator SHALL NOT raise an exception that terminates the run method

### Requirement 18: Configuration Validation

**User Story:** As a system administrator, I want configuration validation at startup, so that I receive clear error messages for misconfiguration before attempting code generation.

#### Acceptance Criteria

1. WHEN get_llm_client is called, THE Configuration_Module SHALL verify OPENAI_API_KEY is set in environment
2. IF OPENAI_API_KEY is not set, THE Configuration_Module SHALL raise a ValueError with message "OPENAI_API_KEY environment variable is not set"
3. WHEN Configuration_Module loads settings, THE Configuration_Module SHALL verify CHROMA_DB_PATH directory exists
4. IF CHROMA_DB_PATH does not exist, THE Configuration_Module SHALL raise a FileNotFoundError indicating the RAG database is not initialized
5. THE Configuration_Module SHALL validate MODEL_NAME is a non-empty string
6. THE Configuration_Module SHALL validate MAX_TOKENS is a positive integer

### Requirement 19: Prompt Template Loading

**User Story:** As a prompt engineer, I want prompt templates loaded from external files, so that I can modify prompts without changing code.

#### Acceptance Criteria

1. WHEN Dev_Agent initializes, THE Dev_Agent SHALL verify prompts/dev_agent_prompt.txt exists
2. WHEN QA_Agent initializes, THE QA_Agent SHALL verify prompts/qa_agent_prompt.txt exists
3. IF a prompt template file is missing, THE agent SHALL raise a FileNotFoundError with the expected file path
4. WHEN loading prompt templates, THE agents SHALL read files with UTF-8 encoding
5. WHEN filling prompt templates, THE agents SHALL replace all placeholder variables with actual values
6. IF a placeholder variable is not provided, THE agent SHALL raise a KeyError indicating the missing variable

### Requirement 20: Round-Trip Code Integrity

**User Story:** As a quality assurance engineer, I want to verify that generated code can be saved and loaded without corruption, so that the system maintains code fidelity.

#### Acceptance Criteria

1. FOR ALL generated C++ code, WHEN the code is saved to a file and then read back, THE loaded content SHALL match the original generated code exactly
2. FOR ALL generated metadata, WHEN the metadata is saved to JSON and then parsed back, THE parsed data SHALL contain all original fields with correct values
3. THE file write operations SHALL use UTF-8 encoding to preserve special characters
4. THE file read operations SHALL use UTF-8 encoding to match write encoding
